# TalentSphere Design System

> Documentation status: Current design-system implementation guide for frontend tokens, components, layouts, and interaction patterns.

This guide defines the visual and interaction system for the UI redesign. It is intentionally presentation-focused: business logic, data contracts, permissions, and feature behavior must remain owned by existing services, slices, and routes.

## Design Principles

- Quiet operational UI: dense enough for repeat work, with restrained styling and clear hierarchy.
- One route, one purpose: each page should have a defined workflow owner and avoid duplicating another page's core action.
- Token-first styling: colors, spacing, radii, shadows, focus rings, and surfaces come from shared tokens before page-specific classes.
- Accessible by default: every interactive primitive must provide focus visibility, readable contrast, keyboard reachability, and non-color status cues.
- Performance-aware presentation: skeletons reserve space, long lists paginate or scroll in bounded regions, and transitions are short.

## Tokens

Tokens live in `apps/frontend/src/index.css`; compact extension tokens live in `chrome-extension-project/src/index.css`.

| Token family | Usage |
| --- | --- |
| `--bg-*` | App canvas, muted page bands, elevated panels, inset fields, overlays |
| `--text-*` | Primary copy, supporting copy, muted metadata, inverse text |
| `--border-*` | Default dividers, subtle containers, stronger hover/active borders |
| `--accent-*` | Primary actions, selected navigation, focus highlights, key links |
| `--success`, `--warning`, `--destructive` | Status states with visible text/icon labels |
| `--radius-*` | Controls, cards, modals, pills; cards should normally stay at `--radius-lg` |
| `--shadow-*` | Floating menus and modal panels; avoid heavy shadows on page sections |
| spacing tokens | Use 4px increments; common page gaps are 16px, 20px, 24px, and 32px |

Tailwind color aliases are declared through `@theme` in `index.css` for `accent`, `success`, `warning`, and `destructive` families. Prefer these token-backed utility names over one-off hex values.

Core app and extension text/status foreground tokens must keep at least 4.5:1 contrast against shared surface tokens. `npm run validate:ui-design-system` enforces this for light app, dark app, and extension token sets.

## Layout Rules

- Authenticated pages render inside `ResponsiveLayout`.
- Public auth pages use the shared `AuthShell` frame with an H1-named main landmark, named authentication panel, named alternate-entry navigation, named forms, Account Type group semantics, and a named registration next-step status. Keep login/register focused on configured email credentials and role intent; use safe public copy for auth provider failures; do not expose OAuth or password-reset controls unless they are wired to validated auth behavior.
- Desktop shell uses a persistent sidebar and sticky header. Collapsed desktop sidebar links and footer controls must keep explicit accessible names.
- Mobile shell uses a named slide-over menu plus bottom navigation from the same route registry. The header menu trigger must expose expanded/controlled state for the slide-over.
- Page content uses a constrained readable width by default and can opt into wide tool layouts only for grids, tables, boards, editors, or dashboards.
- Page sections are unframed layouts. Use cards for repeated records, dialogs, and genuinely framed tools.
- Avoid cards inside cards. If grouping is needed inside a card, use borders, dividers, or inset panels.
- Keep primary action groups near the `PageHeader`.

## Typography Rules

- Use Inter with system fallback.
- Do not scale type with viewport width.
- Do not use negative letter spacing in compact panels, cards, sidebars, tables, or controls.
- Page titles use `text-2xl` to `text-3xl`; card titles use `text-sm` to `text-base`.
- Supporting text uses `--text-secondary`; metadata uses `--text-muted`.
- Long words and user-generated text must wrap, clamp, or move into a scroll/disclosure surface.
- Exported `Typography` helpers must preserve native heading/body/metadata semantics, pass caller attributes through, and remain wrapping-safe by default.

## Component Usage

Import shared primitives, typography helpers, and the compatibility `PageTemplate` from `components/shared` when creating new shared or route-level UI. Direct file imports remain supported for existing routes, but the barrel is the documented design-system entrypoint and must only re-export primitives or compatibility aliases; domain workflows, service calls, route ownership, and analytics stay outside the barrel.

| Component | Use for | Rules |
| --- | --- | --- |
| `ResponsiveLayout` | Authenticated app shell | Preserve userless passthrough, expose a keyboard-visible skip link to the application content target, render one route-named application content `main` landmark for authenticated users from `getApplicationContentLabel` in `routeRegistry.ts`, expose shell root/skip-link/main/page `data-ui` markers with matching compatibility `data-slot` markers, keep sidebar/header ownership unchanged, and preserve resize-driven sidebar offsets |
| `PageTemplate` | Compatibility page shells | Render shared page-template shell/content markers, keep the shared `PageHeader` optional, expose a named `main` content landmark, and use caller-provided class hooks only for layout extension |
| `PageHeader` | Page purpose, supporting copy, primary actions | One per page; render shared `data-ui` and compatibility `data-slot` markers for the header, copy block, title row, and action group; bind title and description to the header; keep status badges beside the title; and group page actions under a title-specific action label |
| `Button` | Commands and form actions | Render the shared `data-ui="button"` and compatibility `data-slot="button"` markers, preserve caller-owned button types, use icons for tool commands when available, keep loading icons decorative while busy state and labels remain available, require confirmation for destructive actions, and let long labels wrap while icon buttons keep fixed dimensions |
| `Input` | Text, email, password, and compact form fields | Render shared input-field/input root markers plus label/control/icon/helper/error subpart markers with matching compatibility `data-slot` markers, use visible labels, connect helper/error text to the field, preserve caller-provided descriptions, mark invalid fields with `aria-invalid`, hide decorative icons, and keep validation behavior owned by the route |
| `Card` / `GlassCard` / `AuraCard` | Repeated items, dialogs, framed tools | Render shared card/subpart markers, avoid nesting, use consistent padding and hover only when interactive, keep card shells min-width safe, wrap long titles/descriptions, let footers wrap actions, and keep `AuraCard` as a compatibility export of the shared card primitives rather than a separate implementation |
| `AuraImage` | Product, avatar, logo, and illustrative image slots | Always pass meaningful `alt` text or an empty decorative alt, render shared image/media/loading/fallback markers with matching compatibility `data-slot` markers, preserve caller-owned image handlers, prefer lazy/async defaults for non-critical media, and keep failed-image fallbacks named or hidden to match the image purpose |
| `Tabs` | Switching views inside one workflow | Use for peer views, not navigation to unrelated routes; render shared tablist/trigger/icon markers with matching compatibility `data-slot` markers, expose selected-state metadata, keep tab icons decorative, connect tabs to panels with stable prefixes where panels are present, and keep Arrow/Home/End focus aligned with the selected tab |
| `StatusBarSurface` | Compact system, sync, latency, and security status strips | Render the shared `data-ui` and compatibility `data-slot` markers for the status-bar root plus state/indicator/latency/security/sync/separator subparts, preserve caller-owned labels and defaults, keep icons and separators decorative, and use only for status summary surfaces rather than workflow actions |
| `EmptyState` | No data states | One clear explanation is required, render shared empty-state root/icon/title/description/action `data-ui` markers with matching compatibility `data-slot` markers, expose a named empty-state region, connect description text, keep decorative icons hidden, and use one non-submit recovery action where useful |
| `ErrorBoundary` | Fatal route or shell recovery | Use token-backed safe copy, service-unavailable wording when applicable, a reload action, and decorative/focusless recovery icons; never show raw exception messages in the public UI |
| `Skeleton` | Loading states | Match final dimensions with explicit height, min-height, size, aspect ratio, or equivalent inline style; render the shared `data-ui="skeleton"` and compatibility `data-slot="skeleton"` markers; hide decorative placeholders from assistive technologies; preserve explicit status semantics; and respect reduced motion |
| `Badge` | Status, source, and compact metadata | Render the shared `data-ui="badge"` and compatibility `data-slot="badge"` markers, keep visible text present, use optional descriptions for abbreviated labels, preserve caller-owned list/status roles only when the surrounding surface needs them, and do not rely on color alone |
| `Toggle` | Binary preferences and settings | Render the shared `data-ui` and compatibility `data-slot` markers for the toggle root and switch, bind visible label and description to the switch, provide an explicit aria label when no visible label is present, and keep the switch button non-submit inside forms |
| `Toast` | Transient confirmation, warning, and error notices | Use for short-lived feedback only; persistent degraded or failed content must also appear inline. Toast stacks render shared `data-ui` markers with matching compatibility `data-slot` markers for the stack, item, icon, content, title, message, and dismiss subparts, use the named `Toast notifications` region, toast-type-specific status/alert labels, atomic error alerts, and non-submit notification-specific dismiss labels |
| `AuraModal` | Focused review/confirmation | Render shared modal structure `data-ui` markers with matching compatibility `data-slot` markers, keep action order predictable, hide backdrop/close icons from assistive technologies, keep the close control non-submit, trap focus while open, lock background page scrolling, close on Escape, and restore focus to the opener |

## Navigation Patterns

- `routeRegistry.ts` is the source for sidebar, mobile nav, protected routes, and `CommandSearch` destinations.
- `featureOwnership.ts` is the source-backed IA contract for deciding which route or surface owns each major feature, why that surface adds user-journey value, and whether the feature can merge into another section.
- Primary navigation should include domain owners only: Dashboard, Jobs, Candidates, Learning, Challenges, Network, AI, Messages, Admin.
- Account navigation should include Settings and Profile.
- Dashboard can host only the source-approved domain summaries or explicit handoff links documented in `featureOwnership.ts`; domain forms, review modals, preference editors, and search destinations stay out of Dashboard.
- Domain-specific utilities such as Resume, Career Path, Billing, and Post Job may remain routes. They can appear in `CommandSearch` only as secondary route-discovery destinations recorded by `featureOwnership.ts`; workflow controls stay in the owning route. `npm run test:ia` guards this split so utility routes do not drift into primary navigation silently.
- Sidebar, mobile slide-over, legacy mobile menu, and bottom mobile navigation are route-entry surfaces only. They must preserve route-registry role filtering, active `aria-current` state, explicit icon-only control labels, and distinct navigation landmark names without moving feature actions into navigation.
- Header search is route discovery, not product full-text search.
- Shell search uses `CommandSearch` with a named search landmark, combobox/listbox pattern, live result status, label-ranked results, role filtering, bounded result counts, route label plus route-registry description option labels, and a named visible no-result status. It must remain keyboard reachable and should not become a hidden full-text search without a new contract.
- Header notifications use a named popover region. Reminder, account notification, Mark read, Retry notifications, and Load more actions must stay keyboard reachable, source labels must distinguish account sync from fallback data, degraded fallback states must include retry copy, load/read failures must use fixed safe copy without raw provider strings, failed mark-all persistence must restore unread state, and Escape should close the popover while returning focus to the notification trigger.
- Secondary feature placements must be summaries, links, reviewed handoffs, preference snapshots, or search destinations. They must not duplicate the owner route's full workflow controls.
- Not Found recovery is a wildcard route. Public visitors get auth-entry recovery, while authenticated users get Dashboard plus role-filtered route-registry destinations; do not turn it into a second dashboard or product search surface.

## Interaction Patterns

- Critical changes require explicit review, cancel, and confirm states.
- AI output is a draft until accepted in the destination workflow.
- Fallback, local-only, demo, degraded, inferred, stale, and not-configured states must be visible.
- Retry controls should be near the failed content and should not imply that data changed unless the underlying operation supports it.
- Fatal runtime recovery uses `ErrorBoundary`; it must distinguish generic crashes from service/network failures, preserve custom fallback slots, provide a reload action, and keep internal error messages in logs instead of public copy.
- Toasts are for transient confirmation; persistent status belongs inline.
- Toast stacks avoid the mobile bottom navigation and stay in a bounded scroll region on narrow screens so repeated messages do not cover the main workflow.
- Toast icons are decorative; hide them from assistive technologies and put the notification type in the toast accessible name instead.
- Prefer `components/shared/Toast` for current screens. `context/ToastContext` remains a compatibility bridge for older realtime notification imports and must use a distinct `Realtime toast notifications` region plus the same status/alert, dismiss, and auto-dismiss semantics until it is retired or migrated.
- Composer attachment panels should move focus to the newly revealed input and keep hidden file inputs out of normal tab order when a visible Upload button owns file selection.

## Accessibility Rules

- Preserve visible focus rings with `:focus-visible`.
- Icon-only buttons require `aria-label` or adjacent visible text.
- Async updates use `role="status"` or `role="alert"` where appropriate.
- Dialogs must trap focus, lock background page scrolling, close on Escape, and restore control on close through `AuraModal`.
- Tabs use semantic tab roles and keyboard navigation. Keep each tab label stable and concise.
- Color-coded status must include text or an icon with accessible label.
- Every route-level screen must expose one visible main landmark and one visible page heading. Authenticated shell main landmarks must use route-specific names such as `Dashboard application content`, `Profile Detail application content`, or `Post Job application content`; public/auth recovery screens may use their H1 as the main landmark name.
- Form controls need programmatic labels; placeholders are helper copy, not labels.
- Duplicate navigation landmarks need unique accessible names so screen-reader users can distinguish them. The mobile slide-over uses `Expanded mobile navigation`, while the bottom bar uses `Mobile navigation`.
- Command search, header notifications, mobile navigation, shared tabs, and auth form submission must stay operable without a pointer.

## Performance Rules

- Prefer CSS variables and shared classes over repeated inline style blocks.
- Keep skeletons and empty states lightweight. Skeleton pulse animation must be reduced-motion aware, and every static shared `Skeleton` usage must reserve final layout space before data arrives.
- Use bounded scroll regions for menus, notifications, lists, and command search.
- Avoid adding heavy animation libraries for primitive transitions; use short CSS transitions and respect reduced motion.

## Source Guardrails

`npm run validate:ui-design-system` scans the web app and extension source for legacy visual-system regressions. The validator rejects old Aurora/neon/glass classes, decorative gradient utilities, oversized card/control radii, nonstandard letter-spacing utilities, nonzero literal CSS `letter-spacing` declarations, hard-coded black/white Tailwind foreground/background utilities, ad hoc hard-coded hex and raw `rgb()`/`rgba()`/`hsl()`/`hsla()` color literals outside CSS token definitions or explicit export-token blocks, app/extension core text/status token pairs below 4.5:1 contrast, undefined `var(--token)` references, static shared `Skeleton` usages that do not reserve vertical layout space with a height, min-height, size, aspect ratio, or equivalent inline style, and shared `EmptyState` usages without explanatory descriptions. Use semantic app tokens, extension tokens, explicit standalone export tokens, final-state skeleton dimensions, and described no-data states instead.
